Two-Sided Healthcare Scheduling & Records Platform

A full-stack healthcare platform connecting patients and doctors вҖ” a React Native mobile app for patients and a Next.js web dashboard for doctors, both backed by a single Supabase database with row-level security.

`React Native` `Expo` `Next.js` `TypeScript` `Supabase` `Tailwind CSS`

## рҹ“Ң What is Medic Centre?

Medic Centre is a real-time healthcare coordination platform split across two purpose-built apps that share one backend:

- **Patients** book appointments, upload medical records, and track prescriptions from their phone.
- **Doctors** manage their patient roster, process appointment requests, and issue digital prescriptions from a web dashboard.

Both apps read and write to the same Supabase PostgreSQL database. There's no sync layer, no polling, no duplicated state вҖ” a doctor approving an appointment in the web dashboard is visible to the patient in the mobile app on their next fetch, because both are querying the same source of truth.

```
Patient books appt  вҶ’  Supabase insert  вҶ’  Doctor sees it in dashboard  вҶ’  Doctor approves
       вҶ’  Supabase update  вҶ’  Patient sees "confirmed" status in mobile app
```

## рҹ—„пёҸ Database Design вҖ” Why This Schema

Healthcare data has a natural shape: one person can be a patient, a doctor, or both вҖ” and every clinical record (appointment, prescription, note) needs to point at exactly two people: who it's for, and who's responsible for it.

**Design decisions:**

- **`profiles` extends `auth.users`** rather than duplicating auth logic вҖ” Supabase Auth handles credentials, `profiles` just adds a `role` column and contact info.
- **`appointments` and `prescriptions` both store `patient_id` + `doctor_id`** as separate foreign keys, not a single `participants` array вҖ” this keeps RLS policies simple (`WHERE doctor_id = auth.uid()`) instead of needing array containment checks.
- **`medications` is a child table of `prescriptions`**, not a JSON column вҖ” a doctor can prescribe 1 to N drugs per visit, and a relational table lets the web dashboard render/edit each line item independently.
- **`medical_records` stores file references, not files** вҖ” actual documents live in Supabase Storage; the table just tracks metadata and ownership.

### The 5 Core Tables

| Table | Purpose | Key Relationships |
|---|---|---|
| `profiles` | Extends auth users with role, name, contact info | `role` вҲҲ {`patient`, `doctor`} |
| `appointments` | Booking record between one patient and one doctor | `patient_id`, `doctor_id` вҶ’ `profiles.id` |
| `medical_records` | Patient-uploaded documents | `patient_id` вҶ’ `profiles.id`, file in Storage |
| `prescriptions` | Diagnosis + notes + validity window | `patient_id`, `doctor_id` вҶ’ `profiles.id` |
| `medications` | Line items within a prescription | `prescription_id` вҶ’ `prescriptions.id` |

## рҹ—„пёҸ Database Schema

```sql
-- profiles вҖ” extends Supabase auth users
CREATE TABLE profiles (
  id            UUID PRIMARY KEY REFERENCES auth.users(id),
  full_name     TEXT,
  email         TEXT,
  phone         TEXT,
  role          TEXT NOT NULL CHECK (role IN ('patient', 'doctor')),
  date_of_birth DATE,
  created_at    TIMESTAMPTZ DEFAULT now()
);

-- appointments вҖ” one booking between one patient and one doctor
CREATE TABLE appointments (
  id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  patient_id         UUID REFERENCES profiles(id),
  doctor_id          UUID REFERENCES profiles(id),
  appointment_date   DATE NOT NULL,
  appointment_time   TIME NOT NULL,
  status             TEXT DEFAULT 'pending', -- pending | confirmed | completed | cancelled
  notes              TEXT,
  created_at         TIMESTAMPTZ DEFAULT now()
);

-- prescriptions вҖ” diagnosis + validity window
CREATE TABLE prescriptions (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  patient_id    UUID REFERENCES profiles(id),
  doctor_id     UUID REFERENCES profiles(id),
  diagnosis     TEXT NOT NULL,
  notes         TEXT,
  issued_date   DATE DEFAULT CURRENT_DATE,
  valid_until   DATE NOT NULL
);

-- medications вҖ” line items within a prescription
CREATE TABLE medications (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  prescription_id  UUID REFERENCES prescriptions(id) ON DELETE CASCADE,
  name             TEXT NOT NULL,
  dosage           TEXT,
  frequency        TEXT,
  duration         TEXT,
  instructions     TEXT
);

-- medical_records вҖ” patient-uploaded documents
CREATE TABLE medical_records (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  patient_id    UUID REFERENCES profiles(id),
  file_url      TEXT NOT NULL,
  file_name     TEXT,
  uploaded_at   TIMESTAMPTZ DEFAULT now()
);
```

Full schema with RLS policies: [`docs/schema.sql`](./docs/schema.sql)

- A free [Supabase](https://supabase.com) account
- [Expo Go](https://expo.dev/go) app (for mobile testing)

### Step 1 вҖ” Clone the repo

```bash
git clone https://github.com/Amman352/medic-centre.git
cd medic-centre
```

### Step 2 вҖ” Set up the database

Go to Supabase вҶ’ SQL Editor вҶ’ paste and run the full schema from [`docs/schema.sql`](./docs/schema.sql).

### Step 3 вҖ” Configure mobile environment

Create `mobile/.env.local`:

```
EXPO_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=eyJhbGc...your-anon-key...
```

### Step 4 вҖ” Configure web environment

Create `web/.env.local`:

```
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGc...your-anon-key...
```

### Step 5 вҖ” Run everything

Open two terminals:

```bash
# Terminal 1 вҖ” Mobile app
cd mobile
npm install
npx expo start
```

```bash
# Terminal 2 вҖ” Web dashboard
cd web
npm install
npm run dev
```

- Mobile: scan the QR code with Expo Go
- Web: open [http://localhost:3000/dashboard](http://localhost:3000/dashboard)

## рҹ”Қ How Doctor вҶ” Patient Data Flow Works

```
1. Patient registers on mobile     вҶ’  profiles row created, role = 'patient'
2. Patient books appointment        вҶ’  appointments row created, status = 'pending'
3. Doctor opens web dashboard         вҶ’  queries appointments WHERE doctor_id = current doctor
4. Doctor clicks "Approve"               вҶ’  UPDATE appointments SET status = 'confirmed'
5. Patient refreshes mobile app             вҶ’  sees status: confirmed
6. Doctor issues prescription                  вҶ’  prescriptions + medications rows created
7. Patient views prescriptions screen             вҶ’  sees new prescription instantly
```

No webhooks, no push notifications (yet) вҖ” both clients query Supabase directly, so data is consistent the moment either side refreshes.
